# Agentic Mobile Automation Demo

A LangGraph-orchestrated pipeline that uses Gemini to turn a plain-English
requirement into a full, runnable Selenium/pytest test suite: planning test
cases, inspecting the live UI, generating a Page Object, and generating
executable test scripts.

## How it works

```
Planner  ->  UI Inspector  ->  Page Agent  ->  Executor
```

| Step | Agent | Reads | Writes |
|---|---|---|---|
| 1 | `Planner` | `requirements/login_requirement.txt` | `agents/testcases.xlsx` |
| 2 | `UIInspector` | live browser session | `artifacts/screen.xml` |
| 3 | `PageAgent` | requirement + `screen.xml` | `pages/login_page.py` |
| 4 | `Executor` | requirement + page object + test cases | `tests/test_*.py` |

Each step's output feeds the next step's input. The graph is defined in
`graph/workflow.py` using LangGraph; state (generated test cases, the screen
XML path) is passed between nodes directly rather than only through
side-effect files, and each node's failure is caught and logged with which
agent failed.

> **Note on "mobile":** this demo drives desktop Chrome with mobile
> emulation via Selenium (see `tests/conftest.py`), not a real device or
> emulator via Appium — even though `Appium-Python-Client` is listed as a
> dependency. If you need real-device/emulator coverage, you'll need to
> swap `conftest.py`'s driver setup for an Appium session.

## Project structure

```
.
├── main.py                       # Entry point — runs the full LangGraph pipeline
├── agents/
│   ├── planner.py                # Generates test cases from a requirement (-> testcases.xlsx)
│   ├── ui_inspector.py           # Captures the live screen's XML/DOM
│   ├── page_agent.py             # Generates a Page Object from the requirement + XML
│   └── executor.py               # Generates pytest scripts from test cases + Page Object
├── graph/
│   └── workflow.py                # LangGraph wiring: state, nodes, edges
├── pages/
│   └── login_page.py             # Generated Page Object (regenerated by PageAgent)
├── prompts/                       # Prompt templates used by each agent
├── requirements/
│   └── login_requirement.txt     # Plain-English requirement driving the whole pipeline
├── tests/
│   ├── conftest.py                # Selenium driver fixture (Chrome, mobile emulation)
│   ├── test_*.py                  # Generated test scripts (from Executor)
│   ├── test_planner_utils.py     # Unit tests for Planner's pure helpers
│   └── test_excel_reader.py      # Unit tests for ExcelReader
├── utils/
│   ├── llm.py                     # Shared Gemini client, retries, code-fence stripping
│   ├── excel_reader.py           # Reads testcases.xlsx into dicts
│   ├── framework_reader.py       # Feeds existing framework code to prompts as context
│   └── logger.py                  # Shared logging setup
├── config/
│   ├── config.json                # Device, base URL, credentials, LLM model
│   └── settings.py                # Loads config.json + .env into a Settings class
├── artifacts/
│   └── screen.xml                 # Generated by UIInspector
├── .env.example                   # Template for required environment variables
└── requirements.txt
```

## Prerequisites

- Python 3.10+
- Google Chrome installed locally (the demo drives it via Selenium)
- A [Gemini API key](https://ai.google.dev/) with access to the model set in
  `config/config.json` (`gemini-2.5-flash` by default)

Selenium 4.6+ manages the matching ChromeDriver automatically — you don't
need to install `chromedriver` separately.

## Setup

1. Create and activate a virtual environment:
   ```bash
   python -m venv venv
   source venv/bin/activate   # Windows: venv\Scripts\activate
   ```

2. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```

3. Copy `.env.example` to `.env` and add your key:
   ```bash
   cp .env.example .env
   ```
   ```
   GEMINI_API_KEY=your-gemini-api-key-here
   ```

4. Review `config/config.json` (see [Configuration](#configuration) below).

## Configuration

All non-secret settings live in `config/config.json`, loaded by
`config/settings.py`:

```json
{
  "web": {
    "device": "Pixel 7",
    "base_url": "https://www.saucedemo.com/",
    "username": "standard_user",
    "password": "secret_sauce"
  },
  "llm": {
    "provider": "gemini",
    "model": "gemini-2.5-flash"
  }
}
```

| Field | Used by | Description |
|---|---|---|
| `web.device` | `tests/conftest.py` | Chrome mobile-emulation device profile |
| `web.base_url` | `tests/conftest.py` | URL the driver navigates to on startup |
| `web.username` / `web.password` | generated tests | Login credentials for the target app |
| `llm.model` | `utils/llm.py` | Gemini model used by every agent |

Only `GEMINI_API_KEY` is a secret and belongs in `.env`, never in
`config.json`.

> The `web.username` / `web.password` values above are the public demo
> credentials for [saucedemo.com](https://www.saucedemo.com/). If you point
> this at a different app, don't commit real credentials here — override
> them via environment variables instead.

## Running

Run the full pipeline end-to-end:

```bash
python main.py
```

Or run an individual agent:

```bash
python -m agents.planner        # generate test cases -> agents/testcases.xlsx
python -m agents.ui_inspector   # capture artifacts/screen.xml (opens a real browser)
python -m agents.page_agent     # generate pages/login_page.py
python -m agents.executor       # generate tests/test_*.py from testcases.xlsx
```

Each agent expects the previous step's output to already exist (e.g.
`page_agent` needs `artifacts/screen.xml`; `executor` needs
`agents/testcases.xlsx` and `pages/login_page.py`) and will raise a clear
`FileNotFoundError` telling you which agent to run first if it's missing.

## Output artifacts

| File | Regenerated by | Should I commit it? |
|---|---|---|
| `agents/testcases.xlsx` | `Planner` | No — it's generated output |
| `artifacts/screen.xml` | `UIInspector` | No — it's generated output |
| `pages/login_page.py` | `PageAgent` | Optional — treat as generated, or hand-edit and skip regenerating it |
| `tests/test_*.py` | `Executor` | Optional — same tradeoff as above |

If you want a fix or convention to *survive* regeneration (e.g. "always
catch `TimeoutException`, never a bare `except:`"), encode it as an
instruction in the relevant prompt file under `prompts/`, not as a manual
edit to the generated file — the next run will overwrite manual edits.

## Testing

Run the generated UI tests:

```bash
pytest tests/ -v
```

Run just the framework's own unit tests (no browser or API key required):

```bash
pytest tests/test_planner_utils.py tests/test_excel_reader.py -v
```

Custom pytest markers (`smoke`, `sanity`, `regression`) used by generated
tests are registered in `pytest.ini`.

## Troubleshooting

- **`FileNotFoundError: .../testcases.xlsx not found`** — run
  `python -m agents.planner` first, or run `python main.py` for the full
  pipeline.
- **`FileNotFoundError: .../screen.xml not found`** — run
  `python -m agents.ui_inspector` first; this requires Chrome and a working
  Selenium/ChromeDriver setup.
- **Gemini API errors / rate limits** — `utils/llm.py` retries transient
  failures automatically (3 attempts, exponential backoff). Persistent
  failures usually mean an invalid `GEMINI_API_KEY` or an unsupported model
  name in `config/config.json`.
- **Generated code has syntax errors** — `utils/llm.py` strips
  ` ```python ` code fences from responses, but there's no `ast.parse()`
  validation step yet (see [Known limitations](#known-limitations)). If a
  generated file looks malformed, check for leftover markdown from the LLM
  and consider adding validation before writing the file.

## Known limitations

- Single flow (login) is hardcoded via `requirements/login_requirement.txt`;
  extending to other flows means adding new requirement files and prompts.
- No validation step confirms generated pytest scripts are syntactically
  correct before being written to `tests/`.
- `config/config.json` ships with plaintext test credentials for the public
  saucedemo.com demo site only — do not follow this pattern for real
  credentials; use environment variables instead.
- "Mobile" testing is Chrome mobile emulation, not a real device/emulator
  via Appium (see the note in [How it works](#how-it-works)).
